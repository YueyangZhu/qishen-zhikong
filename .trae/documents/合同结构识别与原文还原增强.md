# 合同结构识别与原文还原增强

## 摘要

针对用户反馈的两个核心问题进行修复：
1. **合同结构识别不准 + 顺序乱 + 左栏空间未利用全（出现多余滚动条）**
2. **原文区域未 100% 还原原稿 + 顶部出现"文档前部分"干扰文字**

经 Phase 1 探索确认：用户上传的是 **PDF 文件**，干扰文字为"文档前部分文字"（合同正文条款开始前的标题/甲乙方/封面信息等，被当作普通正文渲染造成干扰）。

---

## 当前状态分析（根因）

### 根因 A：PDF 解析对"文档前部分"识别不足
- 文件：`backend/app/services/pdf_service.py`
- `_detect_type` 中 title 识别条件过严：`idx == 1 and not clause_no and len(first_line) <= 30`
  - 真实 PDF 第一段常包含"合同标题 + 编号 + 日期"等多行内容，首行可能 > 30 字符 → 识别为 body
  - 导致合同标题被当作普通正文段渲染，混入原文区顶部
- header 识别正则 `HEADER_PATTERN` 要求"甲方"后紧跟"（：: "，但真实合同可能有"甲方："或"甲方（采购方）："等格式 → 部分甲乙方段识别失败
- 无封面页/目录页检测：PDF 第一页若为封面（仅标题+少量信息），会被全部提取为段落，混入正文

### 根因 B：段落切分未按甲乙方行切分
- `_split_paragraphs` 按空行切块，块内按条款编号行切分
- 但合同首部"甲方：xxx\n乙方：xxx"若在同一块（无空行分隔），会被合并成一段
- 合并后的长段既不是 title 也不是 header，识别为 body，造成干扰

### 根因 C：左栏布局导致"下方空白 + 滚动条"
- 文件：`src/pages/ReviewDetailPage.tsx`（第 434-498 行）
- Card 高度固定 `calc(100vh - 200px)`，body `flex: 1`，章节列表 `flex: 1, overflow: 'auto'`
- 当合同结构识别不准时，sections 数量异常（每段成节），列表过长 → 出现滚动条
- 同时 Card 固定高度撑满，列表内容未填满时视觉上"下方有空位置"
- 核心矛盾：固定高度 + 内容动态 → 要么空白要么滚动条

### 根因 D：章节边界判定产生冗余 section
- `_split_paragraphs` 中，body 段按 clause_no 变化切节
- 若 PDF 解析出大量无 clause_no 的零散段落（如甲乙方信息后的补充说明），每段都可能因 clause_no=None 而归入"合同首部"或产生异常节
- 导致左栏章节目录与真实合同结构不符

### 根因 E：前端兜底识别与后端规则需同步
- `src/utils/logic.ts` 的 `inferParagraphType` 和 `buildSectionsFromParagraphs` 需与后端增强后的规则保持一致

---

## 修改方案

### 改动 1：增强 PDF 解析（`backend/app/services/pdf_service.py`）

**目标**：准确识别"文档前部分"内容，正确标注 title/header，过滤封面页干扰，保证章节顺序与原稿一致。

**具体修改**：

1. **增强 `_detect_type` 的 title 识别**：
   - 放宽首行长度限制：`idx == 1 and not clause_no` 时，若首行 ≤ 40 字符且含"合同/协议/书/契约"关键词 → title
   - 若第一段是多行（如封面页），取首行作为标题，其余行作为 header 段单独切出

2. **增强 `HEADER_PATTERN` 正则**：
   - 支持"甲方："、"甲方："、"甲方（采购方）："、"甲方:"等更多格式
   - 现有正则已较全，补充允许"甲方"后直接跟中文冒号或英文冒号（当前已含，需验证）

3. **新增封面页检测**：
   - 第一页若文本行数 ≤ 8 且含"合同/协议"关键词 → 视为封面页
   - 封面页内容：首行作为 title 段，其余行作为 header 段（甲乙方信息）或过滤（纯元数据如"合同编号：xxx"）

4. **增强段落切分 `_split_paragraphs`**：
   - 块内除按条款编号切分外，增加按"甲方/乙方"行切分
   - 即：当前行匹配 HEADER_PATTERN 且已有累积内容 → 起始新段落
   - 确保甲乙方信息各自独立成段，便于识别为 header

5. **章节边界优化**：
   - 连续多个无 clause_no 的 body 段（如甲乙方后的补充说明）合并到当前首部节，不各自成节
   - 确保签署段之后的零散段落归入"签署落款"节

**不改**：保留现有页眉页脚过滤逻辑（已生效）、签署段识别逻辑（已生效）。

### 改动 2：前端兜底识别同步（`src/utils/logic.ts`）

**目标**：`inferParagraphType` 和 `buildSectionsFromParagraphs` 与后端增强后的规则保持一致。

**具体修改**：
1. `inferParagraphType`：放宽 title 识别（首行 ≤ 40 字符 + 含"合同/协议"关键词）
2. `HEADER_PATTERN` 同步增强（如有调整）
3. `buildSectionsFromParagraphs`：连续无 clause_no 的 body 段合并到当前节，不产生冗余 section

### 改动 3：左栏布局优化（`src/pages/ReviewDetailPage.tsx`）

**目标**：消除"下方空白 + 多余滚动条"问题，让章节列表合理利用空间。

**具体修改**（第 434-498 行）：
1. Card 高度从固定 `calc(100vh - 200px)` 改为 `maxHeight: 'calc(100vh - 200px)'`，移除固定 `height`
   - 内容少时 Card 自适应高度，不出现空白
   - 内容多时撑满到 maxHeight + 滚动条
2. 章节列表容器 `flex: 1, overflow: 'auto'` 保持，但因 Card 高度自适应，列表不会出现"空白 + 滚动条"矛盾
3. 或者：保持 Card 固定高度，但章节列表项样式优化（减少 marginBottom、紧凑布局），让更多内容可见

**决策**：采用方案 1（maxHeight 自适应），更简洁，彻底解决空白问题。三栏高度不一致可接受（中栏原文区本就需要固定高度滚动）。

### 改动 4：原文渲染优化（`src/features/review/ContractTextView.tsx`）

**目标**：确保 title/header 段被正确差异化渲染，不混入正文造成干扰。

**具体修改**：
1. title 段渲染（已实现）：大字号居中加粗 —— 验证识别正确后即可生效
2. header 段渲染（已实现）：小字号灰色背景左边框 —— 验证识别正确后即可生效
3. 无需大改，依赖改动 1/2 的识别增强

**不改**：body 段渲染逻辑（clauseNo + 风险高亮）、签署段渲染逻辑。

## 假设与决策

1. **"文档前部分文字"= 合同标题 + 甲乙方信息**（非纯垃圾内容），应通过差异化渲染消除干扰感，而非过滤删除。
2. **PDF 封面页**：若检测到第一页为封面（行数少 + 含合同关键词），提取标题行作为 title 段，其余甲乙方行作为 header 段，过滤纯元数据行。
3. **左栏高度**：改为 maxHeight 自适应，接受三栏高度不一致。
4. **不改动样例合同数据**（`sampleContracts.ts`）：样例合同段落已有 type 字段或可被正确识别，问题集中在真实上传 PDF。
5. **不改动演示合同数据**（`contractText.ts`）：已含完整 type 字段和 sections。

---

## 验证步骤

1. **后端验证**（`backend/`）：
   - 启动后端服务
   - 上传一份真实 PDF 合同（含封面/标题/甲乙方/正文条款）
   - 调用 `/api/parse` 接口，检查返回的 paragraphs：
     - 第一段 type 应为 'title'（合同标题）
     - 甲乙方信息段 type 应为 'header'
     - 正文条款段 type 为 'body'
     - 签署段 type 为 'signature'
   - 检查 sections 顺序与原稿一致，无冗余节

2. **前端验证**：
   - 上传同一份 PDF，发起 AI 审核
   - 进入详情页：
     - 左栏：章节目录按原稿顺序展示，无冗余节；Card 高度自适应，无多余空白/滚动条
     - 中栏：原文区顶部为 title 段（居中大字号）+ header 段（灰色背景），不混入正文；正文条款按顺序渲染
     - 点击左栏章节，中栏滚动到对应段落

3. **编译验证**：
   - `npx tsc --noEmit` 通过
   - Python 语法检查通过

4. **回归验证**：
   - 样例合同、演示合同详情页仍正常显示
   - 风险高亮、双向定位功能正常

---

## 风险与兼容

1. **旧任务数据**：已解析的文档存储在 IndexedDB/localStorage，type 字段可能为空 → 前端 `inferParagraphType` 兜底识别已覆盖。
2. **PDF 排版多样性**：启发式规则无法覆盖所有 PDF 排版，对极端复杂排版效果有限 → 在合理范围内增强，不追求 100%。
3. **样例合同**：已有 type 字段，不受影响。
