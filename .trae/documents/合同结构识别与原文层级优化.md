# 合同结构识别与原文层级优化

## 一、问题总结

用户反馈两个核心问题：

1. **合同结构识别不准 + 左栏空间未利用**：左栏"合同结构"区域内容不对应原稿，顺序错乱；下方还有空白就出现滚动条。
2. **原文区开头干扰文字**：原文区不是 100% 还原原稿，上部分出现一堆"识别到"的文字（纯纯干扰）。

## 二、根因分析（基于代码探索）

### 问题1：左栏合同结构识别不准 + 空间未利用

**根因 A（最核心）**：左栏根本没使用 `parsedDoc.sections`，而是用 `risks` 数组按 `paragraphId` 分组反推。
- 位置：[ReviewDetailPage.tsx:447-481](file:///d:/TRAE_work_demo/zhinenghetong/src/pages/ReviewDetailPage.tsx)
- 后果：**没风险的条款在左栏完全缺失**；显示的是"风险段落分布"而非"合同章节目录"。

**根因 B**：样例合同回退路径返回 `sections: []`。
- 位置：[reviewService.ts:699](file:///d:/TRAE_work_demo/zhinenghetong/src/services/reviewService.ts)
- 后果：即使左栏改用 sections，样例合同也拿不到章节。

**根因 C**：左栏写死 `maxHeight: 320, overflow: 'auto'`，外层 div 不参与父级 flex 拉伸。
- 位置：[ReviewDetailPage.tsx:432](file:///d:/TRAE_work_demo/zhinenghetong/src/pages/ReviewDetailPage.tsx)
- 后果：内容多就出滚动条，内容少下方留白；与中栏 `calc(100vh - 200px)` 高度不协调。

### 问题2：原文区开头干扰文字

**根因 D**：`ContractParagraph` 类型无 `type` 字段，无法区分标题段/首部段/正文段/签署段。
- 位置：[types/index.ts:69-75](file:///d:/TRAE_work_demo/zhinenghetong/src/types/index.ts) + [backend/app/schemas/review.py:20-26](file:///d:/TRAE_work_demo/zhinenghetong/backend/app/schemas/review.py)

**根因 E**：ContractTextView 对所有段落统一渲染，仅靠 `clauseNo` 是否存在决定样式。
- 位置：[ContractTextView.tsx:72-131](file:///d:/TRAE_work_demo/zhinenghetong/src/features/review/ContractTextView.tsx)
- 后果：首段合同标题、第二/三段甲乙方信息被当普通正文段以 14px 渲染，无视觉层级，看起来像"开头一堆干扰文字"。

**根因 F**：工具栏标题取 `paragraphs[0]?.text`，正文区又重复渲染第一段，导致标题在工具栏+正文区重复出现。
- 位置：[ContractTextView.tsx:198](file:///d:/TRAE_work_demo/zhinenghetong/src/features/review/ContractTextView.tsx)

**根因 G**：后端 `_parse_pdf` 未过滤页眉/页脚/页码/水印，PDF 解析结果第一段可能混入非正文内容。
- 位置：[pdf_service.py:63-79](file:///d:/TRAE_work_demo/zhinenghetong/backend/app/services/pdf_service.py)

## 三、决策（基于探索结果采用最合理方案）

- **原文渲染**：段落化 + 层级区分（不嵌入 PDF 预览）。保留风险高亮定位能力，通过段落 type 字段差异化渲染解决开头干扰文字。
- **左栏内容**：完整章节目录。基于 `parsedDoc.sections` 渲染所有条款（含无风险条款），风险数量用小标签角标。

## 四、修改方案

### 4.1 类型层：ContractParagraph 加 type 字段

**文件**：[src/types/index.ts](file:///d:/TRAE_work_demo/zhinenghetong/src/types/index.ts)

修改 `ContractParagraph` 接口，新增可选 `type` 字段：
```typescript
export interface ContractParagraph {
  id: string;
  index: number;
  text: string;
  clauseNo?: string;
  clauseTitle?: string;
  /** 段落类型：title=合同标题 header=首部甲乙方信息 body=正文条款 signature=签署落款 */
  type?: 'title' | 'header' | 'body' | 'signature';
}
```

**文件**：[backend/app/schemas/review.py](file:///d:/TRAE_work_demo/zhinenghetong/backend/app/schemas/review.py)

`ContractParagraph` Pydantic 模型同步加 `type` 字段（Optional，默认 None，向后兼容）：
```python
class ContractParagraph(BaseModel):
    id: str
    index: int
    text: str
    clauseNo: Optional[str] = None
    clauseTitle: Optional[str] = None
    type: Optional[str] = None  # title / header / body / signature
```

### 4.2 后端解析层：识别段落类型 + 过滤页眉页脚

**文件**：[backend/app/services/pdf_service.py](file:///d:/TRAE_work_demo/zhinenghetong/backend/app/services/pdf_service.py)

**改动1：`_parse_pdf` 增加页眉/页脚/页码过滤**

页眉页脚识别启发式（满足任一即过滤）：
- 纯数字行（页码）
- 匹配 `^第\s*\d+\s*页` / `^page\s*\d+` 的行
- 在多页中重复出现且长度 < 30 的行（页眉页脚特征）

实现：先收集所有页文本，统计每行在多少页出现，重复率 > 50% 且短行视为页眉页脚过滤掉。

**改动2：`_split_paragraphs` 标注段落 type**

在构建 `ContractParagraph` 时按以下规则标注 type：
- 第一段（idx=1）且无 clauseNo 且文本长度 < 30 → `type='title'`
- 段落文本以 `甲方`、`乙方`、`甲方（`、`乙方（`、`供方`、`需方`、`发包方`、`承包方` 开头 → `type='header'`
- 段落文本以 `签署`、`签字`、`盖章`、`签订日期`、`签订地点`、`本合同一式` 开头，或末尾含 `签字（盖章）` → `type='signature'`
- 其余有 clauseNo 的段落 → `type='body'`
- 无 clauseNo 的非首段 → 默认 `type='body'`（向后兼容）

**改动3：`_detect_clause` 首部段也作为独立章节**

当前所有非条款段落都归到"合同首部"一节。改为：
- title 段单独成节（章节标题=合同标题，clauseNo='标题'）
- header 段（甲乙方）单独成节（章节标题='合同主体'，clauseNo='首部'）
- signature 段单独成节（章节标题='签署落款'，clauseNo='签署'）

这样左栏 sections 能完整反映合同结构：标题 / 合同主体 / 第一条...第N条 / 签署落款。

### 4.4 前端渲染层：ContractTextView 差异化渲染

**文件**：[src/features/review/ContractTextView.tsx](file:///d:/TRAE_work_demo/zhinenghetong/src/features/review/ContractTextView.tsx)

**改动1：ParagraphItem 按 type 差异化渲染**

```tsx
const paraType = para.type ?? inferTypeFallback(para);
if (paraType === 'title') {
  // 大字号 18-20px、居中、加粗、上下 margin 大、无 clauseNo 行
  return <div style={{ textAlign: 'center', fontSize: fontSize + 4, fontWeight: 700, margin: '24px 0 16px' }}>{para.text}</div>;
}
if (paraType === 'header') {
  // 小字号 13px、灰色背景 #fafbfc、padding 8px 12px、无 clauseNo 行
  return <div style={{ fontSize: fontSize - 1, color: COLORS.textSecondary, background: '#fafbfc', padding: '6px 12px', borderRadius: 4, marginBottom: 8 }}>{para.text}</div>;
}
if (paraType === 'signature') {
  // 小字号 13px、灰色背景、斜体或浅色
  return <div style={{ fontSize: fontSize - 1, color: COLORS.textSecondary, background: '#fafbfc', padding: '8px 12px', borderRadius: 4, marginTop: 16 }}>{para.text}</div>;
}
// body 类型：保留现有渲染逻辑（clauseNo + text + 高亮）
return <ExistingParagraphItem ... />;
```

`inferTypeFallback`：当 para.type 为空时的前端兜底识别（与后端规则一致），保证旧数据也能正常显示。

**改动2：工具栏标题不再重复**

工具栏标题改为取 `parsedDoc.title`（由 ReviewDetailPage 传入），或取第一个 type='title' 段落的 text。正文区不再重复渲染 title 类型段（已在 ParagraphItem 中按 title 样式渲染，工具栏标题与正文区标题不重复——工具栏是固定小字号 strong，正文区是大字号居中）。

实际处理：工具栏标题保持显示 `parsedDoc.title`（小字号 strong），正文区 title 段按大字号居中渲染。两者样式差异明显，不算重复。但若用户仍觉重复，可将工具栏标题改为"合同正文"+ 段数 + 风险数，不显示具体标题文字。

**决策**：工具栏左侧改为显示"合同正文"+ 段数 + 风险数（去掉 paragraphs[0]?.text），避免与正文区 title 段重复。`parsedDoc.title` 不在工具栏显示。

const riskCountByPara = useMemo(() => {

## 七、风险与兼容

- `type` 字段为 Optional，旧数据（无 type）由前端 `inferTypeFallback` 兜底识别，向后兼容。
- 已存储的 ParsedDocument（数据库 documents 表）无 type 字段，前端兜底识别能正常显示。
- 后端 `_parse_pdf` 页眉页脚过滤为启发式，极端情况可能误过滤；若发现误过滤，可调整重复率阈值。
- 左栏改用 sections 后，若某些场景 sections 为空（如后端解析失败降级），需有 Empty 兜底。
