# 后端接入说明（阶段2：真实 AI 能力接入）

> 本文档说明如何在本地启动 FastAPI 后端，配置 DeepSeek API，并打通前端真实上传 → 解析 → AI 审核的完整链路。
>
> 适用版本：阶段2（Demo MVP + 真实 AI）。第一阶段前端 Mock 流程不受影响，未启动后端时自动降级 Mock。

## 一、整体架构

```
┌──────────────────────┐      HTTP/JSON       ┌──────────────────────────┐
│  前端 Vite Dev Server │  ─────────────────▶  │  FastAPI 后端（8000）    │
│  localhost:5173/5174 │                       │  - POST /api/parse       │
│  （React + TS）       │  ◀─────────────────  │  - POST /api/extract-fields │
└──────────────────────┘   统一 ApiResponse    │  - POST /api/review-risks│
                                                 │  - GET  /health         │
                                                 └────────┬─────────────────┘
                                                          │ openai SDK
                                                          ▼
                                                 ┌──────────────────────┐
                                                 │  DeepSeek API        │
                                                 │  api.deepseek.com    │
                                                 └──────────────────────┘
```

### 数据流

1. 前端 `ReviewNewPage` 用户手动上传 PDF/DOCX（非样例合同）。
2. 点击「开始 AI 审核」→ 调用 `runFullAIReview`：
   - **解析阶段**：`POST /api/parse` → pdfplumber/python-docx 提取段落。
   - **抽取阶段**：`POST /api/extract-fields` → DeepSeek 抽取 15 个字段（含置信度）。
   - **审核阶段**：`POST /api/review-risks` → DeepSeek 识别 13 类风险，返回结构化 JSON。
3. 前端 `reviewService.createTaskWithAIResult` 将 AI 结果写入 localStorage（任务直接进入 `pending_business` 待人工确认状态）。
4. 跳转 `ReviewDetailPage` 进行人工处理（接受/编辑/忽略/转人工复核）。

### 模式切换（Mock vs 真实）

| 触发条件 | 模式 | 行为 |
|---------|------|------|
| `.env` 中 `DEEPSEEK_API_KEY` 为空，或 `MOCK_MODE=true` | Mock | 后端返回预设的演示字段与风险，便于离线演示 |
| `.env` 中 `DEEPSEEK_API_KEY` 已配置，且 `MOCK_MODE=false` 或 `auto` | 真实 AI | 调用 DeepSeek API 真实解析与审核 |
| 前端检测到后端不可达（`/health` 超时） | 前端降级 | 真实上传场景会报错提示，样例合同仍可走 Mock |

> `MOCK_MODE=auto`（默认）：API Key 未配置 → Mock；已配置 → 真实。

## 二、快速启动

### 1. 创建 .env 配置

```bash
cd backend
copy .env.example .env
```

编辑 `backend/.env`，填入 DeepSeek API Key：

```env
DEEPSEEK_API_KEY=sk-你的真实key
DEEPSEEK_MODEL=deepseek-chat
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
BACKEND_PORT=8000
FRONTEND_ORIGIN=http://localhost:5173,http://localhost:5174
AI_TIMEOUT=120
MOCK_MODE=auto
```

> API Key 申请：https://platform.deepseek.com/ → 创建 API Key → 复制 sk- 开头的字符串。

### 2. 启动后端

**方式一：双击 `backend\run.bat`**（自动创建 venv + 安装依赖 + 启动）

**方式二：命令行**

```bash
cd backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

启动成功后控制台会显示：

- 真实 AI 模式：`✓ 真实 AI 模式已启用，模型：deepseek-chat`
- Mock 模式：`⚠ 当前为 Mock 模式（DEEPSEEK_API_KEY 未配置），AI 接口将返回预设数据`

### 3. 验证后端

打开浏览器访问：

- **API 文档（Swagger）**：http://localhost:8000/docs
- **健康检查**：http://localhost:8000/health

`/health` 返回示例：

```json
{
  "status": "ok",
  "is_mock": false,
  "model": "deepseek-chat",
  "reason": null
}
```

### 4. 启动前端

```bash
npm run dev
```

默认地址 http://localhost:5173/（如已被占用会自动切到 5174）。

## 三、端到端验证

### 验证步骤

1. **登录系统**（业务人员账号）。
2. 进入「新建审核」页面（`/reviews/new`）。
3. 第一步「上传合同」：
   - **不要选择样例合同**，直接拖拽上传一份真实的 PDF/DOCX 合同文件。
4. 第二步「填写信息」：填写合同要素（合同名称、相对方、金额、审核重点等）。
5. 第三步「确认发起」：
   - 顶部 Alert 提示「检测到真实上传文件，将调用后端 AI（DeepSeek）进行真实解析与审核」。
   - 点击「开始 AI 审核」。
6. 弹出进度 Modal，依次显示「正在解析合同文档...」→「正在抽取字段...」→「正在 AI 审核风险...」→「审核完成」。
7. 自动跳转到审核详情页（`/reviews/{taskId}`），三栏布局展示：
   - 左栏：合同原文（真实解析的段落）。
   - 中栏：风险卡片列表（真实 AI 识别的风险项）。
   - 右栏：审核摘要与操作区。
8. 处理完风险后提交法务复核 → 生成审核报告。

### 验证要点

- ✅ 合同原文是用户上传文件的真实内容（不是 DEMO_PARSED_DOCUMENT）。
- ✅ 抽取字段包含 AI 置信度（与 Mock 15 字段不同）。
- ✅ 风险卡片原文可在合同原文中高亮定位。
- ✅ 审核记录显示「AI 系统（DeepSeek）」操作者。
- ✅ 任务列表中该任务为「待人工确认」状态。

## 四、API 端点参考

### `POST /api/parse`

解析上传的合同文件，返回结构化段落。

**请求**：`multipart/form-data`，字段 `file`（PDF/DOCX/TXT，≤10MB）。

**响应**：

```json
{
  "success": true,
  "data": {
    "title": "XX采购合同",
    "sections": [{"id": "s1", "title": "合同标的", "clauseNo": "第二条", "paragraphIds": ["p5", "p6"]}],
    "paragraphs": [{"id": "p1", "index": 0, "text": "...", "clauseNo": "第一条", "clauseTitle": "合同主体"}],
    "fullText": "..."
  },
  "message": "解析成功：12 段"
}
```

### `POST /api/extract-fields`

抽取合同关键字段（15 个字段）。

**请求**：JSON body `{"paragraphs": [...]}`。

**响应**：

```json
{
  "success": true,
  "data": {
    "fields": [
      {"fieldKey": "contractName", "fieldLabel": "合同名称", "fieldValue": "XX采购合同", "confidence": 0.95, "lowConfidence": false, "sourceText": "合同正文第一行"}
    ]
  }
}
```

### `POST /api/review-risks`

AI 审核合同风险（13 类风险类型）。

**请求**：

```json
{
  "paragraphs": [...],
  "contractType": "软件采购",
  "myRole": "buyer",
  "reviewFocus": ["subject", "payment", "breach"],
  "reviewNote": "重点关注付款节点"
}
```

**响应**：

```json
{
  "success": true,
  "data": {
    "risks": [
      {
        "title": "付款节点前置风险",
        "riskType": "payment",
        "riskLevel": "high",
        "clauseNumber": "第四条",
        "clauseTitle": "付款方式",
        "originalText": "...",
        "paragraphId": "p8",
        "startPosition": 12,
        "endPosition": 88,
        "riskReason": "付款比例...",
        "reviewBasis": "依据《xxx》...",
        "suggestion": "建议改为...",
        "confidence": 0.92,
        "sourceType": "ai"
      }
    ],
    "aiSummary": "本次共识别 X 项风险，其中高风险 X 项..."
  }
}
```

### `GET /health`

健康检查 + 当前模式。

**响应**：

```json
{"status": "ok", "is_mock": false, "model": "deepseek-chat", "reason": null}
```

## 五、前端配置

### 环境变量

前端默认连接 `http://127.0.0.1:8000`，如需修改，在项目根目录创建 `.env.local`：

```env
VITE_API_BASE=http://your-backend-host:8000
```

### 关键文件

| 文件 | 作用 |
|------|------|
| `src/services/apiClient.ts` | 前端 API 客户端，封装 parse/extract/review 三个端点 + `runFullAIReview` 完整流程 |
| `src/services/reviewService.ts` | `createTaskWithAIResult` 将 AI 结果写入 localStorage；`getDocument` 优先读取真实解析结果 |
| `src/services/db.ts` | 新增 `documents` 集合存储真实 AI 解析的合同文档 |
| `src/pages/ReviewNewPage.tsx` | 根据 `sampleId` 与 `file.rawFile` 判断走真实 AI 还是 Mock 流程 |

### 判断逻辑

```typescript
// ReviewNewPage.handleStartReview
const useRealAI = !sampleId && !!file?.rawFile;
```

- 选择样例合同 → Mock 流程（创建草稿 → 启动 → 跳进度页）。
- 手动上传文件 → 真实 AI 流程（解析 → 抽取 → 审核 → 直接进详情页）。

## 六、故障排查

### 问题 1：前端点击「开始 AI 审核」报错「Failed to fetch」

**原因**：后端未启动，或地址不对。

**解决**：
1. 确认后端 `uvicorn` 进程正在运行（访问 http://localhost:8000/health）。
2. 检查前端 `.env.local` 的 `VITE_API_BASE` 配置。
3. 检查浏览器控制台是否有 CORS 错误，确认 `backend/.env` 的 `FRONTEND_ORIGIN` 包含前端地址。

### 问题 2：后端日志显示 `401 Unauthorized` / `invalid api key`

**原因**：DeepSeek API Key 无效或已过期。

**解决**：
1. 登录 https://platform.deepseek.com/ 检查 API Key 状态。
2. 重新生成 Key 并更新 `.env`。
3. 确认账户余额充足。

### 问题 3：AI 审核超时（默认 120 秒）

**原因**：大模型响应较慢，特别是复杂合同。

**解决**：
1. `backend/.env` 增大 `AI_TIMEOUT=300`（5 分钟）。
2. 前端 `src/services/apiClient.ts` 的 `DEFAULT_TIMEOUT` 同步调大。
3. 检查网络是否能访问 `api.deepseek.com`。

### 问题 4：解析 PDF 返回段落为空

**原因**：PDF 可能是扫描件（图片型 PDF），文本层为空。

**解决**：当前 MVP 不支持 OCR，请使用：
- 文字型 PDF（可复制文字）。
- 或 DOCX 格式。
- 阶段3 计划接入 OCR（暂未实现）。

### 问题 5：审核结果字段对不上

**原因**：DeepSeek 返回的 JSON 字段名可能不一致。

**解决**：
1. 访问 http://localhost:8000/docs 用 Swagger 测试。
2. 查看后端日志中的原始 AI 响应。
3. 如有字段映射问题，修改 `backend/app/services/prompt_service.py` 的提示词约束。

## 七、Mock 模式说明

Mock 模式适用于：
- 没有 DeepSeek API Key 时的离线演示。
- CI/自动化测试。
- 后端未启动时前端开发联调。

Mock 模式下：
- `/api/parse` 返回固定演示合同段落。
- `/api/extract-fields` 返回固定字段集。
- `/api/review-risks` 返回固定风险列表（与前端 Mock 一致）。

> Mock 模式的输出与真实 AI 输出格式完全一致，便于切换。

## 八、安全注意事项

1. **不要将 `.env` 提交到 Git**（`.gitignore` 已排除）。
2. **不要在前端代码中硬编码 API Key**（应通过后端中转）。
3. **生产环境部署**：阶段2 仅供本地演示，不部署线上（参见 AGENTS.md）。
4. **API Key 轮换**：定期到 DeepSeek 平台轮换 Key。
