# AGENTS.md — 契审智控｜AI采购合同审核平台 项目协作规则

> 本文件是本项目的长期开发规则与上下文说明。任何 Agent（AI 或人类）在修改本项目代码前，必须先阅读本文件。

## 一、项目定位

- **产品名称**：契审智控｜AI采购合同审核平台
- **产品定位**：面向企业采购业务的 AI 合同风险审核工作台，覆盖「上传 → 解析 → 字段抽取 → 规则检查 → AI 语义审核 → 原文定位 → 修改建议 → 人工确认 → 法务复核 → 审核报告 → 留档」核心闭环。
- **当前阶段**：第一阶段 —— 本地可交互前端 Demo 与完整业务闭环（作品集 MVP）。
- **最高依据**：`reference/契审智控_AI采购合同审核平台_PRD_V1.0.docx`（PRD V1.0）。
- **演示主流程**：见 `docs/开发计划.md`「演示主流程」一节，必须可端到端跑通。

## 二、技术栈与约束

- React 18 + TypeScript（严格模式）+ Vite 5
- Ant Design 5（唯一 UI 框架，禁止混用其他 UI 库）
- React Router 6（数据持久化后刷新可访问）
- Zustand（状态管理）
- ECharts + echarts-for-react（图表）
- dayjs（时间）
- lucide-react / @ant-design/icons（图标）
- localStorage 轻量 Mock 数据层（第一阶段，不接真实后端）

### 禁止事项
- 不接入真实大模型 API / 真实 Supabase / 真实鉴权
- 不部署线上、不推送 GitHub、不修改项目目录外文件
- 不实现扫描 PDF OCR、Word 红线修订、电子签章、复杂审批流
- 不把所有代码放一个文件、不大量使用 `any`、不硬编码重复样式与状态值
- 不擅自扩展到销售/劳动合同场景
- 不使用 `alert/confirm` 原生弹窗替代完整交互
- 不使用 Lorem ipsum、英文模板残留、虚假按钮、静态图片冒充页面


## 四、统一界面规范（必须遵守）

### 风险等级映射（全局唯一，禁止各页面自定义）
| 值 | 中文 | 色值 | 含义 |
|----|------|------|------|
| high | 高风险 | #f5222d（红） | 必须人工确认，未处理不得生成报告 |
| medium | 中风险 | #fa8c16（橙） | 建议处理 |
| low | 低风险 | #52c41a（绿） | 可批量处理 |
| notice | 提示项 | #8c8c8c→蓝灰 | 不阻断流程 |

### 审核任务状态（全局唯一）
`draft` 草稿 / `parsing` 解析中 / `ai_reviewing` AI审核中 / `pending_business` 待人工确认 / `pending_legal` 待法务复核 / `completed` 已完成 / `failed` 失败

### 风险处理状态（全局唯一）
`pending` 待处理 / `accepted` 已接受 / `edited` 已编辑 / `ignored` 已忽略 / `manual_review` 转人工复核 / `confirmed` 已确认

### 配色规范
- 主色：专业蓝 `#1677ff`
- AI 辅助色：青绿 `#13c2c2`
- 背景：浅灰白 `#f5f7fa`
- 卡片圆角适中、阴影轻微，禁止大面积渐变与科技光效

### 通用规范
- 主色蓝 + AI 青绿辅助；浅色背景；企业级 B 端 SaaS 风格
- 1366×768 核心内容可用，三栏审核页不出现横向滚动
- 所有按钮三种归宿：真实执行 / 跳真实页面 / 明确提示「当前版本暂未开放」
- 关键操作：表单校验、成功/失败提示、二次确认、加载态、防重复提交、空/错状态
- 表单防抖 300ms；列表筛选写 URL；危险操作二次确认

## 五、数据一致性原则（核心红线）

1. **统一 Mock 数据层 + 业务服务层**：所有页面通过 `services/*` 读写数据，禁止页面各自编造。
2. **同一任务数据贯穿**：工作台 / 列表 / 详情 / 法务 / 报告 / 记录共享同一数据源。
3. **联动规则**：
   - 详情处理风险 → 列表风险数量同步变化
   - 提交法务复核 → 任务状态 `pending_business → pending_legal`
   - 法务完成审核 → 生成对应报告
   - 报告重大风险 = 最终确认的风险
   - 审核记录展示真实发生过的操作
   - 工作台统计从统一数据源实时计算
4. **持久化**：localStorage 持久化，刷新后状态不丢失。
5. **报告数据快照**：报告来自「已提交的审核快照」，后续修改不静默改变已生成报告。

## 六、AI 审核能力（前端 Mock 规则）

- AI 结果必须为结构化 JSON（见 `docs/数据模型.md` RiskItem），不接受 Markdown 作为正式结果。
- 每项风险必须含 `originalClause` 原文，无法定位则降置信度并默认进人工复核。
- 置信度策略：≥0.85 正常；0.60–0.84 建议复核；<0.60 默认转人工。
- 不展示模型内部思维过程，仅展示风险结论、触发规则、原文证据、简要说明。
- **AI 免责声明（必须在所有页面与报告展示）**：
  > 本系统审核结果由 AI 辅助生成，仅供合同初审参考，不构成正式法律意见，最终结论应由专业人员确认。

## 七、提交法务复核前置校验

- 所有 `high` 风险均已处理（accepted/edited/ignored/manual_review/confirmed）后才允许提交
- 无未保存编辑
- 合同基本信息已确认
- 存在未处理风险时提交需二次确认

## 八、代码与提交规范

- TypeScript 严格模式，组件职责清晰，页面/业务组件/通用组件/服务/类型分层
- 代码注释使用中文
- 不创建无关文档；不擅自删除有效文件
- 仅在用户明确要求时创建 git commit，且需本地测试通过后再提交
- 用户偏好：中文沟通、step-by-step 指导、变更需本地验证后再推送 GitHub

## 九、预留真实后端接口

`services/*` 中所有方法命名与 `docs/API设计.md` 一致，第一阶段为 Mock 实现，后续替换为 FastAPI + Supabase + 大模型 API 时只需改实现不改调用方。

## 十、本阶段不实现（但需有明确提示）

- 真实文件解析（DOCX/PDF）→ Mock 解析进度
- 真实大模型审核 → Mock 15+ 风险
- 真实 PDF 导出 → 浏览器打印实现
- Word 导出 → 未实现时明确提示，不伪造成功
- 电子签章、履约管理、复杂审批流
